01

Khái niệm ảo hóa (Virtualization)

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Đặt vấn đề: Triển khai truyền thống

Hạn chế của máy chủ vật lý (Bare-metal)

Vấn đề

Cài trực tiếp lên phần cứng

Một máy chủ vật lý chạy duy nhất một hệ điều hành và phục vụ ứng dụng thô dẫn đến nhiều điểm nghẽn.

  • Lãng phí tài nguyên: Chỉ sử dụng 10% - 15% năng lực thật sự của CPU/RAM.
  • Triển khai chậm chạp: Mua sắm, đi dây, lắp rack mất nhiều tuần.
  • Thiếu tính độc lập: Xung đột dependency khi chạy nhiều app.
© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Công nghệ ảo hóa là gì?

Trừu tượng hóa tài nguyên vật lý

Ảo hóa (Virtualization) là việc chèn một lớp phần mềm trung gian lên trên phần cứng vật lý để phân tách nó thành các máy ảo (Virtual Machine - VM) hoàn chỉnh chạy độc lập.

Lịch sử

Khởi nguồn từ 1967 (CP-40 trên Mainframe IBM) và phát triển bùng nổ bởi VMware từ năm 1998.

Máy ảo (Guest OS)
Lớp ảo hóa (Hypervisor)
Phần cứng vật lý (Host)
© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Thành phần cốt lõi: Hypervisor

Trình quản lý máy ảo (VMM)

  • Chức năng: Trừu tượng hóa phần cứng vật lý, phân phối CPU/RAM/Disk linh hoạt cho các VM.
  • Bảo mật: Đảm bảo tính cô lập (isolation) giữa các máy ảo, tránh truy cập chéo bộ nhớ.
  • Quyền điều khiển (CPU Rings): CPU phân chia các quyền xử lý từ Ring 0 (cao nhất) tới Ring 3 (thấp nhất). Hypervisor quản lý & dịch các lệnh nhạy cảm của Guest OS thông qua Binary Translation hoặc phần cứng hỗ trợ (Intel VT-x, AMD-V).
© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Hypervisor Type 1 vs Type 2

Phân loại cấu trúc ảo hóa

Type 1 (Bare-metal)

  • Cài trực tiếp lên phần cứng vật lý.
  • Hiệu năng tiệm cận phần cứng thực tế.
  • Phù hợp cho Cloud & Data Center.
  • Ví dụ: KVM, VMware ESXi, Hyper-V.

Type 2 (Hosted)

  • Chạy trên một hệ điều hành nền (Host OS).
  • Hiệu năng kém hơn do qua lớp OS trung gian.
  • Phù hợp cho môi trường Dev/Test cá nhân.
  • Ví dụ: VirtualBox, VMware Workstation.

02

Khởi tạo Cloud Server thực tế

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Đặt vấn đề: Triển khai trên Cloud

Đáp ứng nhu cầu tài nguyên tức thời

Trong phát triển hiện đại, kỹ sư DevOps không cần mua và lắp đặt máy chủ vật lý. Thay vào đó, ta sử dụng **Cloud Server** để cấp phát tài nguyên ảo hóa linh hoạt chỉ trong vài phút qua Internet.

Khái niệm

Virtual Private Server (VPS): Là một máy ảo được phân tách từ máy chủ vật lý lớn, chạy hệ điều hành, tài nguyên độc lập và sở hữu IP tĩnh riêng.

Thông số cấu hình của Cloud Server

Các chỉ số phần cứng cần lưu ý khi tạo VPS

vCPU (Virtual CPU)

Đơn vị xử lý ảo hóa trung tâm.

RAM (Random Access Memory)

Khi thiếu RAM, HĐH sẽ kích hoạt cơ chế OOM-Killer tắt các app.

Storage

Sử dụng ổ SSD/NVMe ảo hóa để tăng tốc IOPS đọc/ghi dữ liệu.

Region / Zone

Vị trí vật lý của DC. Nên chọn gần user nhất (VD: Singapore).

Cơ chế xác thực bằng SSH Key

Mật mã học khóa công khai (Public-key Cryptography)

SSH Key giúp đăng nhập an toàn, vô hiệu hóa hoàn toàn nguy cơ bị dò quét brute-force mật khẩu truyền thống.

  • Private Key (`id_ed25519`): Giữ bí mật trên máy tính local.
  • Public Key (`id_ed25519.pub`): Lưu trữ trên Cloud VM để xác thực quyền kết nối.

Thuật toán

Sử dụng Ed25519 thay thế cho RSA do hiệu năng cao hơn, bảo mật tốt và tệp khóa ngắn gọn.

Cơ chế bảo mật cho Infrastructure

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Đặt vấn đề: Mối đe dọa trực tuyến

Tại sao cần gia cố bảo mật máy chủ ngay lập tức?

Một máy chủ thô khi vừa online có địa chỉ IP công cộng sẽ liên tục bị các công cụ tự động (botnet) dò quét cổng và mật khẩu.

Nguy cơ

Mở tất cả các cổng và dùng mật khẩu đăng nhập yếu sẽ khiến máy chủ bị hack và lợi dụng đào coin chỉ sau vài giờ.

  • Tài khoản root mặc định: Dễ bị brute-force đoán password.
  • Không có Firewall: Lộ diện các tiến trình nội bộ ra môi trường mạng internet.
© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

SSH Hardening (Gia cố SSH)

Hạn chế tối đa bề mặt tấn công qua cổng 22

  • Tạo non-root user và chỉ nâng quyền qua `sudo`.
  • Vô hiệu hóa password: Buộc sử dụng SSH Key.
  • Tắt login root trực tiếp: Kẻ tấn công không thể chiếm quyền quản trị tối cao ngay từ bước đầu.
/etc/ssh/sshd_config config 
PermitRootLogin no
PasswordAuthentication no

Cơ chế hoạt động của Firewall

Bộ lọc kiểm soát lưu lượng mạng ra vào

Default Deny Incoming

Mặc định chặn toàn bộ các gói tin đi vào. Chỉ cho phép các cổng dịch vụ được khai báo tường minh hoạt động.

  • Cổng 22 (SSH): Chỉ mở cho IP của quản trị viên (hoặc mở rộng có kiểm soát).
  • Cổng 80 (HTTP) & 443 (HTTPS): Mở rộng rãi cho người dùng truy cập web.
  • Default Allow Outgoing: Cho phép máy chủ tự do kết nối ra ngoài để cập nhật hệ thống.

Thực hành cấu hình UFW trên Ubuntu

Uncomplicated Firewall - Tường lửa tích hợp

Các lệnh khai báo thiết lập quy tắc tường lửa căn bản trước khi kích hoạt:

  • 1. Cho phép cổng SSH.
  • 2. Cho phép cổng Web (HTTP/HTTPS).
  • 3. Kích hoạt UFW.
ufw_setup.sh bash 
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status verbose

Cài đặt và cấu hình Web Server Nginx

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Đặt vấn đề: Phân phối dịch vụ Web

Tại sao cần một Web Server trung gian?

  • Hạn chế App Server thô: Các ứng dụng Node.js/Python không được tối ưu để phân phối file tĩnh (HTML, CSS, Image) cho hàng nghìn request cùng lúc.
  • Bảo mật: Phơi bày trực tiếp cổng ứng dụng nội bộ (như 3000, 5000, 8080) ra ngoài Internet tăng nguy cơ bị tấn công dịch vụ.
  • Quản lý: Không thể chạy đồng thời nhiều trang web chung một cổng HTTP (80) tiêu chuẩn.
© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Nginx Web Server

Kiến trúc hướng sự kiện (Event-driven) chịu tải cao

Nginx (Engine X) là phần mềm mã nguồn mở hoạt động hiệu quả như một Web Server phục vụ nội dung tĩnh, Reverse Proxy điều phối request và Load Balancer.

  • Port 80: HTTP (Không mã hóa thông tin).
  • Port 443: HTTPS (Mã hóa bảo mật TLS/SSL).

Ưu điểm

Chiếm cực ít tài nguyên RAM/CPU so với Apache truyền thống khi chịu tải lượng truy cập đồng thời lớn.

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Cấu trúc thư mục Nginx trên Ubuntu

Các file và thư mục quản trị cần nhớ

  • /etc/nginx/nginx.conf: File cấu hình tổng của dịch vụ Nginx.
  • /etc/nginx/sites-available/: Nơi lưu các file cấu hình Server Block của từng website.
  • /etc/nginx/sites-enabled/: Chứa các liên kết tượng trưng (symlinks) trỏ sang sites-available để Nginx biên dịch hoạt động.
  • /var/www/html/: Thư mục gốc chứa source code tĩnh (HTML/CSS) mặc định.

Cấu hình Server Block (Virtual Host)

Chạy nhiều website khác nhau trên cùng một IP máy chủ

Nginx phân biệt các request gửi đến dựa trên cấu hình tên miền `server_name` và điều hướng truy cập.

  • `listen 80`: Cổng HTTP.
  • `root`: Thư mục chứa code.
  • `index`: File ưu tiên thực thi.
/etc/nginx/sites-available/myweb nginx 
server {
    listen 80;
    server_name myweb.com;
    root /var/www/myweb;
    index index.html;
}

Bảo mật HTTP Basic Authentication

Yêu cầu đăng nhập trực tiếp trên trình duyệt

Bảo vệ tài nguyên nội bộ bằng cách cấu hình Nginx yêu cầu tài khoản/mật khẩu được băm trước đó.

  • Sử dụng lệnh `htpasswd` của gói `apache2-utils` để băm và tạo file mật khẩu.
  • Chỉ thị `auth_basic` hiển thị thông báo.
  • Chỉ thị `auth_basic_user_file` trỏ đến file tài khoản.
nginx_auth.conf nginx 
location /secure/ {
    auth_basic "Restricted Area";
    auth_basic_user_file /etc/nginx/.htpasswd;
}
