01

Khái niệm ảo hóa (Virtualization)

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Vấn đề của máy chủ Bare-metal

Cài OS và app trực tiếp lên phần cứng — vì sao không còn đủ?

Hạn chế

  • Lãng phí tài nguyên: một app thường chỉ dùng khoảng 10–15% CPU/RAM thật, nhưng vẫn phải trả điện, làm mát và khấu hao cả máy.
  • Triển khai chậm: mua sắm, lắp rack, đi dây và cài tay có thể mất nhiều tuần.
  • Thiếu isolation: nhiều app chung máy dễ xung đột dependency; một app sập hoặc bị tấn công có thể kéo theo cả host.

Nhu cầu mới

  • Chia một máy vật lý mạnh thành nhiều “máy ảo” độc lập.
  • Cấp phát nhanh hơn (phút/giờ thay vì tuần).
  • Tách môi trường Dev / Staging / Prod an toàn hơn trên cùng hạ tầng.

Ảo hóa là gì?

Trừu tượng hóa phần cứng thành nhiều máy ảo độc lập

Virtualization chèn một lớp phần mềm trung gian lên phần cứng vật lý để trừu tượng hóa CPU, RAM, disk và network thành nhiều phần độc lập. Mỗi phần đóng gói thành một .

Lịch sử ngắn

1967 — CP-40 trên IBM Mainframe. 1998 — VMware đưa ảo hóa bùng nổ trên x86. Sau đó có tăng tốc phần cứng Intel VT-x / AMD-V.

Hypervisor (VMM)

Thành phần phân phối tài nguyên và cô lập các VM

  • Phân phối tài nguyên: cấp CPU/RAM/Disk linh hoạt cho từng Guest OS.
  • Isolation: không cho VM này đọc bộ nhớ hay disk của VM khác một cách trái phép.
  • CPU Rings: lệnh nhạy cảm của Guest được dịch (Binary Translation) hoặc hỗ trợ phần cứng .

Ghi nhớ

Không có Hypervisor ổn định thì không có Cloud VPS đáng tin — đây là lớp “xương sống” giữa phần cứng thật và máy ảo bạn thuê.

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Type 1 vs Type 2

Bare-metal hypervisor · Hosted hypervisor

Type 1 · Bare-metal

Cài trực tiếp lên phần cứng. Hiệu năng & isolation cao — chuẩn Cloud. Ví dụ: KVM, ESXi, Hyper-V.

Type 2 · Hosted

Chạy trên Host OS có sẵn. Dễ cài laptop, overhead cao hơn. Ví dụ: VirtualBox, VMware Workstation.

Trong lab Cloud

VPS (DigitalOcean, AWS, …) gần như luôn nằm trên stack Type 1 (thường KVM) tại Data Center provider.

Cloud Server / VPS thay thế bare-metal

Cấp phát máy ảo qua Internet trong vài phút

  • Không cần mua rack, IP tĩnh vật lý hay cấu hình router tại nhà/công ty.
  • Provider đã ảo hóa sẵn host Type 1; bạn chọn OS, region, kích thước máy và bấm Create.
  • = slice ảo độc lập: OS, CPU/RAM/disk, IP tĩnh — cô lập với VPS hàng xóm.

Xác thực bằng SSH Key

Thay password — chống brute-force ngay từ bước tạo máy

  • Private key (id_ed25519): chỉ nằm trên máy bạn — không commit, không chat.
  • Public key (id_ed25519.pub): dán lên provider / ~/.ssh/authorized_keys.
  • Ưu tiên thuật toán Ed25519: ngắn, nhanh, bảo mật hiện đại hơn RSA dài dòng.
generate_key.sh bash 
ssh-keygen -t ed25519 -C "you@rikkei.academy"
# Gắn public key khi tạo Droplet, rồi:
ssh root@YOUR_VPS_IP

03

Bảo mật cho Infrastructure

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Vì sao phải harden ngay khi online?

IP public = bề mặt tấn công trong vài phút

  • Botnet quét cổng 22 (SSH), panel mặc định và password yếu 24/7.
  • Giữ root + PasswordAuthentication dễ bị brute-force tới khi đoán trúng.
  • Không firewall = mọi process lắng nghe đều lộ ra Internet.

Nguy cơ thực tế

Máy “thô” (unhardened) có thể bị chiếm quyền, đào coin hoặc phát tán spam chỉ sau vài giờ có IP public. Hardening SSH + UFW là bước bắt buộc, không phải optional.

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

SSH Hardening

Thu hẹp bề mặt tấn công cổng 22

  1. 01

    User thường + sudo

    Không làm việc hằng ngày bằng root.

  2. 02

    Tắt password login

    Chỉ còn public-key.

  3. 03

    Tắt root SSH

    PermitRootLogin no

/etc/ssh/sshd_config config 
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
# sudo systemctl reload ssh

An toàn khi sửa

Giữ một phiên SSH đang mở khi test cấu hình mới — tránh tự khóa ra khỏi máy.

Web Server Nginx

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Vì sao cần Nginx?

Không phơi app thô ra Internet

App server thô

  • Node/Python kém tối ưu file tĩnh với hàng nghìn kết nối.
  • Phơi cổng 3000/5000/8080 tăng bề mặt tấn công.
  • Khó nhiều site chung cổng chuẩn 80/443.

Có Nginx

  • Event-driven, tốn ít RAM/CPU khi serve static.
  • Lớp biên: TLS, auth, (sau này) reverse proxy.
  • Server Block = nhiều website / một IP.

Luồng session này

Client → cổng 80/443 → Nginx → file tĩnh dưới /var/www/…. Reverse proxy app Java/Node thuộc session sau — không làm trong buổi này.

© 2026 By Rikkei Academy - Rikkei Education - All rights reserved.

Cấu trúc Nginx và Server Block

Path Ubuntu · virtual host cơ bản

  • /etc/nginx/nginx.conf — cấu hình tổng
  • /etc/nginx/sites-available/ — file từng site
  • /etc/nginx/sites-enabled/ — symlink site đang bật
  • /var/www/html/ — document root mặc định
listen
Cổng (80 / 443)
server_name
Tên miền / Host header
root · index
Thư mục code · file mặc định
/etc/nginx/sites-available/myweb nginx 
server {
    listen 80;
    server_name myweb.com;
    root /var/www/myweb;
    index index.html;
}

Basic Auth và phạm vi session

Bảo vệ path nội bộ · không loãng sang deploy app

nginx_auth.conf nginx 
location /secure/ {
    auth_basic "Restricted Area";
    auth_basic_user_file /etc/nginx/.htpasswd;
}

Tạo user băm mật khẩu: sudo apt install apache2-utils rồi sudo htpasswd -c /etc/nginx/.htpasswd admin.

Tóm tắt mang về

Checklist kiến thức và kỹ năng

  • Bare-metal lãng phí và chậm; ảo hóa + Hypervisor Type 1 là nền Cloud VPS.
  • Tạo VPS: chọn region/spec, gắn SSH Key Ed25519, đăng nhập không password yếu.
  • Hardening: user + sudo, tắt root/password SSH; UFW mở 22/80/443 rồi enable.
  • Nginx: sites-available/enabled, Server Block, optional Basic Auth cho path nội bộ.
